Conceptos Fundamentales de Ingeniería de Software

Resumen de principios y fundamentos que todo desarrollador debe dominar.

Principios ACID

Atomicidad

Una transacción se ejecuta completamente o no se ejecuta. No se permiten estados intermedios.

Consistencia

Garantiza que las reglas del sistema (restricciones, integridad, etc.) se mantengan antes y después de la transacción.

Aislamiento

Las transacciones concurrentes no interfieren entre sí, evitando efectos de lectura o escritura no deseados.

Durabilidad

Una vez confirmada, la transacción persiste incluso ante fallos del sistema.

Principios SOLID

Single Responsibility

Una clase debe tener una única razón para cambiar. Promueve cohesión y claridad.

Open/Closed

Las entidades deben estar abiertas a extensión pero cerradas a modificación. Se logra mediante interfaces y herencia.

Liskov Substitution

Los objetos de una subclase deben poder reemplazar a los de su superclase sin alterar la funcionalidad del programa.

Interface Segregation

Las interfaces deben ser específicas y no forzar a implementar métodos innecesarios.

Dependency Inversion

Las dependencias deben estar definidas en abstracciones, no en implementaciones concretas.

Pilares de la Programación Orientada a Objetos

Abstracción

Oculta los detalles innecesarios mostrando solo la información esencial para el usuario del objeto.

Encapsulamiento

Protege el estado interno del objeto y controla su acceso mediante métodos públicos.

Herencia

Permite crear nuevas clases basadas en otras, promoviendo reutilización y jerarquías lógicas.

Polimorfismo

Permite que un mismo método o interfaz se comporte de diferentes maneras según el objeto que lo implemente.

Otros Conceptos Clave

DRY (Don't Repeat Yourself)

Evita la duplicación de lógica o información. Centraliza la responsabilidad para facilitar el mantenimiento.

KISS (Keep It Simple, Stupid)

El diseño debe ser lo más simple posible, evitando la complejidad innecesaria.

YAGNI (You Aren’t Gonna Need It)

No implementes funcionalidad que no se necesita aún; prioriza lo que aporta valor inmediato.

Principio de Responsabilidad Única

Relacionado con SOLID, recalca que cada módulo debe encargarse de una parte claramente delimitada del sistema.

Principio de Inversión de Dependencias

Los módulos de alto nivel no deben depender de los de bajo nivel; ambos deben depender de abstracciones.

Consejo: Comprender estos principios es clave para diseñar sistemas mantenibles, escalables y robustos.
