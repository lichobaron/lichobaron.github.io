Resumen: Patrones de Diseño en POO

Patrones comunes agrupados por categoría, con cuándo usarlos y pros/cons.

Clásicos • Sobrio • Responsive

Patrones Creacionales

Singleton

Garantiza una única instancia de una clase y un punto global de acceso. Útil para recursos compartidos (configuración, logs). Evitar abuso: dificulta pruebas y genera acoplamiento global.

Factory Method

Define una interfaz para crear un objeto, pero deja que las subclases decidan qué clase instanciar. Favorece extensión sin modificar código cliente.

Abstract Factory

Proporciona una interfaz para crear familias de objetos relacionados sin especificar clases concretas. Útil para variantes de plataformas o temas.

Builder

Separa la construcción compleja de un objeto de su representación, permitiendo construir objetos paso a paso. Ideal para objetos con muchas configuraciones.

Prototype

Crea nuevos objetos clonando una instancia prototipo. Útil cuando la creación es costosa o se requiere duplicar estados complejos.

Patrones Estructurales

Adapter

Convierte la interfaz de una clase en otra que el cliente espera. Útil para integrar librerías o APIs incompatibles sin cambiar el código existente.

Bridge

Desacopla una abstracción de su implementación, de modo que ambas puedan variar independientemente. Bueno para plataformas cruzadas.

Composite

Compone objetos en estructuras de árbol para representar jerarquías parte-todo. Permite tratar objetos individuales y compuestos de la misma forma.

Decorator

Agrega responsabilidades adicionales a un objeto de forma dinámica. Alternativa flexible al uso de subclases para extender comportamiento.

Facade

Proporciona una interfaz unificada y simplificada a un conjunto de subsistemas. Reduce la complejidad para el cliente y mejora el acoplamiento.

Flyweight

Minimiza el uso de memoria compartiendo objetos que son iguales internamente. Útil cuando hay muchas instancias similares.

Proxy

Proporciona un sustituto o representante de otro objeto para controlar el acceso (lazy loading, seguridad, logging).

Patrones Conductuales

Strategy

Define una familia de algoritmos intercambiables y los encapsula para que sean intercambiables en tiempo de ejecución.

Observer

Define una dependencia uno-a-muchos entre objetos; cuando uno cambia, sus dependientes son notificados.

Command

Encapsula una petición como un objeto, permitiendo parametrizar clientes con operaciones, soportar undo/redo y colas de tareas.

Iterator

Proporciona una forma de acceder secuencialmente a los elementos de una colección sin exponer su representación interna.

Mediator

Define un objeto que encapsula la forma en que interactúan un conjunto de objetos, reduciendo las dependencias directas entre ellos.

Memento

Permite capturar y restaurar el estado interno de un objeto sin violar su encapsulación. Útil para snapshots y undo.

State

Permite a un objeto alterar su comportamiento cuando cambia su estado interno.

Template Method

Define el esqueleto de un algoritmo en una operación, delegando algunos pasos a subclases.

Visitor

Representa una operación a realizar sobre los elementos de una estructura de objetos. Permite agregar operaciones sin modificar las clases de los elementos.

Chain of Responsibility

Pasa una petición a lo largo de una cadena de objetos hasta que uno la maneja.

Consejos rápidos:
  • Comprende el problema antes de aplicar un patrón.
  • Evita sobreingeniería: usa patrones donde aporten valor.
  • Prioriza claridad, mantenibilidad y testabilidad.
