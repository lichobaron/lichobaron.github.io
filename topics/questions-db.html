Preguntas de Entrevista: JPA, Hibernate y Bases de Datos

1. JPA (Java Persistence API)

¿Qué es JPA y cuál es su propósito?

JPA es una especificación de Java que define cómo mapear clases Java a tablas de bases de datos relacionales. Simplifica el manejo de datos al permitir trabajar con objetos en lugar de SQL directamente.

¿Cuál es la diferencia entre EntityManager y Session?

EntityManager pertenece a JPA (interfaz estándar), mientras que Session es específica de Hibernate. Ambos gestionan el ciclo de vida de las entidades, pero Session ofrece funcionalidades adicionales propietarias de Hibernate.

¿Qué estrategias de generación de IDs existen en JPA?

Las más comunes son:

  • GenerationType.IDENTITY – la base de datos genera el ID (autoincrement).
  • GenerationType.SEQUENCE – usa secuencias (útil en Oracle o PostgreSQL).
  • GenerationType.TABLE – almacena los IDs en una tabla dedicada.
  • GenerationType.AUTO – JPA elige según la base de datos.

¿Cómo maneja JPA el Lazy Loading?

Las relaciones con fetch = FetchType.LAZY cargan los datos solo cuando se acceden. Esto evita traer datos innecesarios, pero puede generar errores de sesión cerrada si se accede fuera de la transacción.

¿Qué son los Entity Graph en JPA y cómo se usan?

Un Entity Graph en JPA es una característica que permite definir qué relaciones deben cargarse de manera explícita cuando se consulta una entidad, optimizando el rendimiento y evitando el N+1 problem.

En lugar de usar fetch = EAGER o consultas personalizadas, puedes definir un Entity Graph que especifique qué asociaciones se deben traer de forma fetch (cargadas inmediatamente) y cuáles deben permanecer perezosas (lazy).

Ejemplo:


// Definición del EntityGraph en la entidad
@Entity
@NamedEntityGraph(
    name = "User.withRoles",
    attributeNodes = {
        @NamedAttributeNode("roles")
    }
)
public class User {
    @Id
    private Long id;
    private String name;

    @OneToMany(mappedBy = "user")
    private List roles;
}

// Uso del EntityGraph en una consulta
EntityGraph graph = entityManager.getEntityGraph("User.withRoles");
Map<String, Object> properties = new HashMap<>();
properties.put("javax.persistence.fetchgraph", graph);

User user = entityManager.find(User.class, 1L, properties);

En este ejemplo, al buscar un User, JPA cargará automáticamente los roles definidos en el Entity Graph sin necesidad de usar un JOIN FETCH en la consulta. Esto mejora la legibilidad del código y te da control dinámico sobre qué relaciones se cargan en distintos contextos.

2. Hibernate

¿Qué papel juega el SessionFactory?

SessionFactory es un objeto pesado y thread-safe que crea y administra sesiones (Session) en Hibernate. Suele instanciarse una sola vez por aplicación.

¿Qué diferencia hay entre persist(), merge() y save()?

  • persist(): Método estándar JPA. Inserta una nueva entidad si no existe.
  • merge(): Sincroniza una entidad detached con la base de datos.
  • save(): Método propio de Hibernate. Retorna el ID generado.

¿Qué son los niveles de caché en Hibernate?

Hibernate maneja dos niveles de caché:

  • Primer nivel: asociada a la sesión, activa por defecto.
  • Segundo nivel: compartida entre sesiones; configurable (Ehcache, Infinispan, etc.).

¿Qué es el N+1 problem y cómo evitarlo?

Ocurre cuando se ejecuta una consulta para una entidad principal y luego se hacen consultas adicionales por cada elemento relacionado. Se evita usando fetch join, EntityGraph o configurando relaciones con FetchType.LAZY.

3. Bases de Datos y Replicación

¿Qué es la replicación de bases de datos?

Es el proceso de mantener copias sincronizadas de una base de datos en varios servidores. Mejora la disponibilidad y el rendimiento en entornos distribuidos.

¿Qué tipos de replicación existen?

  • Maestro-Esclavo: un nodo principal escribe, los demás solo leen.
  • Multi-Maestro: todos los nodos pueden escribir; requiere resolución de conflictos.
  • Asíncrona: las réplicas se actualizan con retardo.
  • Síncrona: la escritura se confirma solo cuando todas las réplicas actualizan.

¿Qué problemas puede causar la replicación?

Puede generar inconsistencias temporales, conflictos de escritura, y aumentar la latencia en escrituras síncronas.

4. Optimización de Consultas

¿Cómo optimizar consultas en Hibernate?

  • Usar JOIN FETCH para evitar N+1 queries.
  • Proyectar solo los campos necesarios con SELECT new.
  • Utilizar batch-size en colecciones grandes.
  • Configurar cachés de segundo nivel.
  • Indexar columnas frecuentemente consultadas.

¿Cómo identificar consultas lentas?

Activando logs SQL en Hibernate (spring.jpa.show-sql=true) y analizando el plan de ejecución (EXPLAIN en SQL).

5. Optimización y Rendimiento de Bases de Datos

¿Qué factores afectan el rendimiento?

  • Falta de índices en columnas clave.
  • Consultas no parametrizadas (provocan recompilación del plan).
  • Transacciones demasiado largas.
  • Bloqueos de tablas o filas.
  • Exceso de joins innecesarios.

¿Cómo mejorar la performance?

  • Usar índices y particiones adecuadamente.
  • Analizar estadísticas y planes de ejecución.
  • Optimizar consultas con agregaciones o subconsultas.
  • Utilizar caching y compresión de resultados.
  • Desnormalizar parcialmente si el caso lo amerita.

6. Modelos Relacionales y No Relacionales

¿Cuáles son las ventajas de los modelos relacionales?

  • Integridad referencial garantizada (mediante claves foráneas).
  • Lenguaje estándar de consultas (SQL).
  • Transacciones ACID confiables.
  • Ideal para datos estructurados y relaciones complejas.

¿Cuáles son las ventajas de los modelos NoSQL?

  • Alta escalabilidad horizontal.
  • Mayor flexibilidad de esquema (documentos, grafos, pares clave-valor).
  • Excelente rendimiento en lectura/escritura masiva.
  • Adecuado para datos no estructurados o en constante cambio.

¿Cuándo conviene usar uno u otro?

Usa relacional cuando requieras consistencia fuerte, transacciones complejas y relaciones entre datos. Usa NoSQL cuando priorices escalabilidad, disponibilidad o esquemas variables.
